## Part 1. このツールは何か（できること・利益）

**AI Security Audit System** は、AI（大規模言語モデル, LLM）を診断の中心に据えた
**自律型セキュリティ診断プラットフォーム**です。従来ツールが「既知パターン（シグネチャ）」
への一致で脆弱性を探すのに対し、本ツールは LLM の**文脈理解・推論**を組み合わせ、
**未知の脆弱性や設計上の論理欠陥**まで踏み込んで探索します。

### 4つの動作モード

| モード | できること | 主な対象 |
|---|---|---|
| **CODE AUDIT** | ソースコードを意味的に解析し、設計上の脆弱性（SQLi・IDOR・競合状態など）を発見 | `.py` ファイル |
| **ATTACK MODE** | ポートスキャン → サービス特定 → Web探査 → AI攻撃仮説 → PoC生成（**生成のみ・送信しない**） | サーバ・Webサービス |
| **WEB FUZZ** | Webをクロールして入力点を抽出し、検出プローブで脆弱性の兆候を観測（**検出のみ**・認証対応） | Webアプリ |
| **DEFENSE MODE** | ログを監視し、攻撃をリアルタイム検知して MITRE ATT&CK で分類 | アクセスログ |

### 設計思想（“何をしないか”が重要）

本ツールは攻撃ツールではなく**診断ツール**です。次の原則を守って実装されています。

- **検出のみ（Detection-only）**: 脆弱性の「発見・兆候観測」に特化。データ窃取・RCE実行・認証回避などのエクスプロイトは**しない**。
- **PoCは生成のみ・送信しない**: ATTACK MODE の PoC は**ローカルで生成・表示・保存するだけ**。対象へ送信/実行する経路を持たず、`EXPLOIT_TRANSMISSION_ENABLED=False` で不変条件を明示。
- **非DoS**: 総リクエスト数に上限、ジッター・低並列で過負荷を回避。
- **ローカル保存**: 結果・PoC・レポートはローカルのみ。外部へ自動送信しない。
- **認可前提**: 許可されたターゲットにのみ使用する（[Part 5](#part-5-セキュリティと法的注意)）。

### 結果として得られる利益

1. **未知の脆弱性に気づける**: シグネチャに無い論理欠陥（認可漏れ・競合状態など）をAIが推論で指摘。
2. **学習コストが低い**: 自然言語で「なぜ危険か・どう直すか」を説明してくれるため、初学者の教材になる。
3. **一気通貫**: 偵察→分析→レポート（HTML/PDF・日英）まで自動。手作業の集計が要らない。
4. **安全に練習できる**: 検出のみ＋同梱の脆弱サイト（[testlab/](../testlab/)）で、合法的に手を動かして学べる。
5. **攻守両面**: 攻撃者視点（ATTACK/FUZZ）と防御者視点（DEFENSE）を同じUIで体験でき、SOC/レッドチーム双方の理解が深まる。

## Part 2. 何をしているか（アーキテクチャと処理の流れ）

### 全体構造

```
GUI(メインスレッド)  ──操作──>  Agent(別スレッド)  ──>  Tools(scanner/prober/fuzzer/...)
        ^                            |                         |
        |                            v                         v
        └────── EventBus(Queue) ── 進捗/出力/集計 ──        LLMClient ──> LLM(API)
```

- **GUI** ([gui/app.py](../gui/app.py)): CustomTkinter 製。4タブ。ユーザー操作を受けてエージェントを起動。
- **エージェント** ([agents/](../agents/)): 各モードの“頭脳”。バックグラウンドスレッドで実行され、ツールを呼び、LLMに推論させる。
- **ツール** ([tools/](../tools/)): 実際の作業（ポートスキャン・HTTP探査・ファジング・ログ追尾・CVE照合・レポート生成）。
- **EventBus** ([core/event_bus.py](../core/event_bus.py)): スレッド間を `queue.Queue` で安全に橋渡し。GUIのスレッド制約を回避する要。
- **LLMClient** ([core/llm_client.py](../core/llm_client.py)): OpenAI互換APIの抽象化（Ollama/OpenAI/OpenRouter 共通）。

### なぜこの設計なのか（ジュニア向けの肝）

- **スレッド分離**: スキャンやLLM応答は時間がかかる。これをGUIスレッドで行うと画面が固まる。
  そこでエージェントを別スレッドにし、結果は **EventBus（キュー）** 経由でGUIに渡す。
  tkinter は別スレッドから直接UIを触れないため、この“キューに積む→GUIが取り出す”が定石。
- **GUI非依存のロジック**: `agents/ tools/ core/` はGUIに依存しない。だから将来Web化（API化）する際も
  バックエンドをそのまま再利用できる（[Part 1の利益](#結果として得られる利益)にも通じる）。
- **基底クラス** ([agents/base_agent.py](../agents/base_agent.py)): 全エージェントが継承。`threading.Event` による
  停止制御（`is_stopped()`）、LLMストリーミング、深刻度のリアルタイム集計、PoC/調査レポートの保存などを共通提供。

### 共通の処理パターン

1. ユーザーがターゲット/ファイルとオプションを入力し実行。
2. エージェントが**情報収集**（コード読込 / ポートスキャン / クロール / ログ取得）。
3. 収集結果を**要約してLLMへ**送り、**ストリーミング**で分析を受け取りUIへ流す。
4. 出力中の `SEVERITY: ...` をリアルタイム集計し **DETECTION SUMMARY** を更新。
5. 結果を **reports/** に保存（HTML/PDF は手動、調査レポート/PoC は自動）。

## Part 3. モード別の詳細（できること／していること／利益）

### CODE AUDIT — コードの意味解析

- **していること** ([agents/audit_agent.py](../agents/audit_agent.py)):
  Pythonソースを読み込み、LLMへ「意図と実装のギャップ」を探させる。出力された `CWE-xxx` から
  NVD API でCVEを照合 ([tools/cve_client.py](../tools/cve_client.py))。`LangGraph` モードでは
  CRITICAL発見時に深層解析ループを回す ([agents/langgraph_audit_agent.py](../agents/langgraph_audit_agent.py))。
- **検出例**: SQLインジェクション、予測可能トークン、TOCTOU競合、タイミング攻撃、IDOR、パストラバーサル、認可ロジック欠陥。
- **利益**: 静的解析ツールが見逃す**論理的欠陥**を、理由と修正案つきで把握できる。

### ATTACK MODE — 自律偵察とPoC生成

- **していること** ([agents/recon_agent.py](../agents/recon_agent.py)):
  1. ホスト解決 → 2. **ポートスキャン**（TCP connect、スコープ `common`/`extended`/`full`、UDP任意）
  → 3. バナーグラブ → 4. **Web探査**（[tools/web_prober.py](../tools/web_prober.py)）
  → 5. **受動OSフィンガープリント**（追加通信なし） → 6. **AI攻撃仮説** → 7. **PoC生成（送信なし）**。
- **ステルス**: プロファイル（`stealth`〜`aggressive`）でジッター・走査順ランダム化・UAローテーション・並列度を制御。
- **PoCの安全制御**: 生成物は表示・ファイル保存のみ。送信経路を持たず、ガード関数 `_assert_no_transmission()` で担保。
- **利益**: 攻撃者の思考（攻撃面の洗い出し→攻撃チェーン→検証PoC）を**安全に追体験**でき、防御の優先度づけに使える。

### WEB FUZZ — スマートファジング（検出のみ・認証対応）

- **していること** ([agents/fuzz_agent.py](../agents/fuzz_agent.py) / [tools/web_fuzzer.py](../tools/web_fuzzer.py)):
  1. （任意）**認証**（Cookie / ヘッダー / ログインフォーム＋CSRF自動抽出）
  → 2. 同一オリジンを浅く**クロール**して注入点（クエリ/フォーム）を抽出
  → 3. AIが**検出プローブ**を生成 → 4. **ファジング**（レスポンス異常を観測） → 5. **AIトリアージ**。
- **検出シグネチャ**: 未エスケープ反射（XSS）、DBエラー署名（SQLi）、`/etc/passwd` 署名（トラバーサル）、`{{7*7}}→49`（SSTI）。
- **利益**: 認証後の画面も含め、**入力処理の弱点の兆候**を網羅的に観測。検出のみなので本番に近い環境でも比較的安全。

### DEFENSE MODE — リアルタイム脅威監視

- **していること** ([agents/monitor_agent.py](../agents/monitor_agent.py)):
  ログを `tail -f` 風に追尾し ([tools/log_watcher.py](../tools/log_watcher.py))、正規表現の
  `ATTACK_PATTERNS` で一次検知 → **ALERT TIMELINE** へ即時表示 → バッチをLLMへ送り
  **MITRE ATT&CK** のTTPで分類・解説。
- **利益**: ブルーチーム視点でのインシデント検知・分類を体験。攻撃の“見え方”を学べる。

## Part 4. 用語・知識リファレンス（勉強用）

> ここからは辞書的に使ってください。各項目は「定義 → 本ツールでの登場箇所」をセットで記載します。

### 4-A. 脆弱性

- **OWASP Top 10**: Webアプリの代表的リスクを10個にまとめた業界標準リスト。本ツールの検出対象の多くが該当。
- **CWE (Common Weakness Enumeration)**: 脆弱性の“種類”を番号で表す辞書（例: CWE-89=SQLi）。CODE AUDITが出力し、CVE照合に使う。
- **CVE (Common Vulnerabilities and Exposures)**: 実在する個別の脆弱性に振られるID（例: CVE-2021-44228）。NVDで管理。
- **NVD (National Vulnerability Database)**: 米国NISTのCVEデータベース。本ツールは API v2 で CWE→関連CVE を照合。

主要な脆弱性クラス（本ツールが検出・言及するもの）:

- **SQLインジェクション (SQLi / CWE-89)**: 入力をSQL文に連結してしまい、DB操作を乗っ取られる。
  検出: 入力に `'` を混ぜたときの**DBエラー署名**（"You have an error in your SQL syntax" 等）。
- **クロスサイトスクリプティング (XSS / CWE-79)**: 入力がHTMLにそのまま出力され、被害者のブラウザで任意JSが動く。
  種類: 反射型 / 格納型 / DOM型。検出: ペイロードが**未エスケープで反射**するか。
- **サーバサイドテンプレートインジェクション (SSTI / CWE-1336)**: テンプレートエンジンに式を評価させる。
  検出: `{{7*7}}` が `49` として評価されるか。
- **パストラバーサル / LFI / RFI (CWE-22/98)**: `../` 等でファイルパスを抜け、想定外ファイルを読む(LFI)／外部を読み込む(RFI)。
  検出: `../../etc/passwd` で `root:x:0:0:` 形式が返るか。
- **コマンドインジェクション (CWE-78)**: 入力がOSコマンドに渡り任意コマンド実行。DEFENSEの `CMD_INJECTION` で監視。
- **IDOR / 不適切なアクセス制御 (CWE-639/284)**: 他人のIDを指定すると他人のデータが見える等、認可の欠落。CODE AUDITの主対象。
- **CSRF (クロスサイトリクエストフォージェリ / CWE-352)**: 利用者の意図しないリクエストを第三者サイトから送らせる。
  対策が**CSRFトークン**。WEB FUZZ の認証フォームではこのトークンを自動抽出して正規にログインする。
- **SSRF (サーバサイドリクエストフォージェリ / CWE-918)**: サーバに内部資源へリクエストさせる。DEFENSEで `http://169.254...` 等を監視。
- **TOCTOU / 競合状態 (Race Condition / CWE-367/362)**: 「確認」と「使用」の間の時間差を突く。CODE AUDITが推論で指摘。
- **タイミング攻撃 (CWE-208)**: 処理時間差から秘密情報を推測（例: 文字列比較の早期リターン）。
- **認証/認可の設計欠陥**: 認証(本人確認)と認可(権限確認)の取り違え・抜け。LLMが文脈から指摘。
- **情報漏洩 / 機微パス露出**: `/.env` `/.git/HEAD` `/admin` `/backup.zip` 等が公開されている状態。`SENSITIVE_PATHS` で探索。
- **セキュリティヘッダの欠如**: 後述（[4-C](#4-c-web技術)）。防御の“付け忘れ”を検出。
- **安全でないCookie**: `Secure` / `HttpOnly` 属性が無く、盗聴やJSからの窃取に弱い。プローバが指摘。
- **ブルートフォース**: 総当たりログイン試行。DEFENSEの `BRUTE_FORCE` で監視（本ツール自身は実施しない）。

### 4-B. ネットワーク／ポート／サービス

- **TCP / UDP**: TCPは接続確立(3wayハンドシェイク)を伴う信頼通信、UDPはコネクションレス。
  本ツールのTCPスキャンは「connectできたら開放」。UDPは無応答だと **open|filtered**（区別不能）になる曖昧さがある。
- **ポートスキャン**: どのポートが開いているか調べる行為。攻撃面の入口を知る基本。
  実装: [tools/network_scanner.py](../tools/network_scanner.py)（`socket` のみ、nmap不要）。
- **スキャンスコープ**: `common`(主要26) / `extended`(約125) / `full`(全65535)。`full` は時間がかかる。
- **バナーグラブ**: 接続後にサーバが返す文字列（ソフト名/バージョン等）を取得し、サービスやCVEの手掛かりにする。
- **受動OSフィンガープリント**: 取得済みバナー/ヘッダーだけからOSを推定（追加パケットを送らない=ステルス）。
  nmap の `-O`（能動探査）とは異なる。
- **代表的なポートとサービス**（`SERVICE_MAP` 抜粋。番号→サービス）:

  | ポート | サービス | ひとこと |
  |---|---|---|
  | 21 | FTP | 平文ファイル転送（危険） |
  | 22 | SSH | 暗号化リモートシェル |
  | 23 | Telnet | 平文リモート（危険） |
  | 25 | SMTP | メール送信 |
  | 53 | DNS | 名前解決 |
  | 80 / 443 | HTTP / HTTPS | Web（443はTLS） |
  | 110/143 | POP3/IMAP | メール受信 |
  | 135/139/445 | RPC/NetBIOS/SMB | Windows系（445は攻撃対象になりやすい） |
  | 1433/1521/3306/5432 | MSSQL/Oracle/MySQL/PostgreSQL | DB |
  | 3389 | RDP | Windowsリモートデスクトップ |
  | 6379 | Redis | インメモリDB（無認証露出に注意） |
  | 8080/8443/8000 | HTTP-Alt 等 | 開発/プロキシでよく使う |
  | 9200 / 27017 | Elasticsearch / MongoDB | 無認証露出事故が多い |
  | 161/udp | SNMP | 機器管理（情報漏洩源になりうる） |

- **SSL/TLS 証明書**: HTTPS通信の暗号化と同一性保証。プローバは CN・発行者・有効期限を取得。
- **IDS / IPS**: 侵入検知/防止システム。スキャンの“静かさ”はこれを意識した話（次項）。

### 4-C. Web技術

- **HTTP/レスポンスステータス**: 200(OK)/301,302(リダイレクト)/401,403(認証/認可)/404(無し)/500(サーバエラー)。
  ファザーは「`'` を送ったら 500 になった」等の**状態変化**を脆弱性の兆候とみなす。
- **HTTPヘッダー**: リクエスト/レスポンスのメタ情報。`User-Agent`/`Cookie`/`Authorization`/`Server`/`X-Powered-By` 等。
- **セキュリティヘッダー**（`SECURITY_HEADERS`。“在るべきもの”）:

  | ヘッダー | 役割 |
  |---|---|
  | Strict-Transport-Security (HSTS) | 常時HTTPSを強制 |
  | Content-Security-Policy (CSP) | 読み込み元を制限しXSSを緩和 |
  | X-Content-Type-Options | MIMEスニッフィング抑止 |
  | X-Frame-Options | クリックジャッキング防止 |
  | X-XSS-Protection | 旧来のXSSフィルタ |
  | Referrer-Policy | リファラ送出の制御 |
  | Permissions-Policy | 端末機能（カメラ等）の許可制御 |

- **Cookie / セッション**: ログイン状態を保持する仕組み。`Secure`(HTTPSのみ送出)/`HttpOnly`(JS不可)/`SameSite`(CSRF緩和)。
- **CSRFトークン**: フォーム等に埋め込む使い捨ての秘密値。正規リクエストの証明に使う。WEB FUZZ認証で自動抽出。
- **CORS / 同一オリジンポリシー**: ブラウザが別オリジンへのアクセスを制限する仕組み（PWA化の壁にもなる）。
- **クロール**: リンクを辿ってページ/入力点を収集する処理。本ツールは同一オリジン限定・浅め。
- **注入点 (injection point)**: ユーザー入力がサーバに渡る箇所（クエリパラメータ・フォーム項目）。ファジングの標的。
- **技術スタック検出**（`TECH_SIGNATURES`）: レスポンスの特徴から使用技術を推定。
  検出対象例: WordPress / Drupal / Joomla / Django / Laravel / React / Angular / Vue / jQuery / Bootstrap /
  ASP.NET / PHP / Ruby on Rails / Express / Spring Boot / Apache / Nginx / IIS / Cloudflare。
- **認証方式**（WEB FUZZ対応）: Cookieセッション / Bearerトークン(ヘッダー) / フォームログイン。

### 4-D. AI／LLM

- **LLM (大規模言語モデル)**: 大量テキストで学習した言語モデル。本ツールでは「コード/ログ/偵察結果の理解と推論」に使う。
- **OpenAI互換API**: `chat/completions` 形式のAPI規格。これに準拠していればプロバイダを差し替え可能。
- **バックエンド例**:
  - **Ollama**: ローカルでLLMを動かす（無料・APIキー不要）。既定の接続先。
  - **OpenAI**: GPT系のクラウドAPI。
  - **OpenRouter**: 1つのキーで多数モデル（Claude/GPT/Qwen等）を使える集約サービス。
- **モデル例**: GPT-5.5 / Claude（Opus 4.8・Sonnet 4.6・Fable 5）/ Qwen3 / Llama 4 Scout など。
  本ツールは接続先の `/v1/models` から一覧を取得（[settings_dialog](../gui/dialogs/settings_dialog.py) の「↻取得」）。
- **ストリーミング**: 応答を少しずつ受け取り即時表示する方式。本ツールはチャンクをEventBus経由でUIへ流す。
- **プロンプト / システムプロンプト**: モデルへの指示文。各エージェントが役割（レッドチーム/SOC等）を定義。
- **トークン**: モデルが扱うテキストの最小単位。入出力量・料金の基準。
- **温度・max_tokens等のパラメータ**: 生成のランダム性や長さの制御値。
- **セマンティック解析**: 構文一致でなく“意味”に基づく解析。本ツールが未知脆弱性を狙える理由。
- **LangGraph**: LLMを状態機械(StateGraph)として組むフレームワーク。CODE AUDITの深層解析ループに利用。
- **レポート言語制御**: 設定の `report_lang` に応じ、AIの出力言語とレポート表記を日/英で切替（base_agent が指示を自動付与）。

### 4-E. 攻撃者・防御者の知識体系

- **レッドチーム / ブルーチーム**: 攻撃側 / 防御側。本ツールは ATTACK/FUZZ が赤、DEFENSE が青に相当。
- **ペネトレーションテスト（ペンテスト）**: 許可の下で実際に攻撃を試み、弱点を実証する診断。
- **MITRE ATT&CK**: 実攻撃の戦術・技術を体系化したナレッジベース。DEFENSEがTTP分類に使用。
- **TTP (Tactics, Techniques, Procedures)**: 攻撃者の戦術・技術・手順。脅威の“やり口”を表す。
- **キルチェーン / 攻撃チェーン**: 偵察→侵入→権限昇格…と続く一連の攻撃段階。ATTACKのAI仮説が組み立てる。
- **PoC (Proof of Concept)**: 脆弱性が“本当に存在する”ことを示す最小の実証コード。本ツールは生成のみ（送信しない）。
- **エクスプロイト**: 脆弱性を実際に突くコード/手法。本ツールは**実行・送信しない**（PoC止まり）。
- **ステルス / IDS回避の基礎**（本ツールのプロファイルで体験できる）:
  - **ジッター**: リクエスト間にランダム遅延を入れ、機械的なレート検知を避ける。
  - **走査順ランダム化**: 連番スキャンはIDSに目立つため順序をシャッフル。
  - **UAローテーション**: 実在ブラウザの `User-Agent` を回し、自己申告型UAでの検知を避ける。
  - **低並列**: 同時接続を抑え“静かに”探る。
- **フットプリント**: 攻撃の“足跡（検知されやすさ）”。stealth=小、aggressive=大。
- **誤検知 (False Positive) / 検知漏れ (False Negative)**: 診断結果の読み解きで常に意識する2種の誤り。

### 4-F. 本ツールの技術スタック

- **Python 3.10+**: 実装言語。
- **CustomTkinter / Tkinter**: デスクトップGUI（ダークテーマ）。`tkinter` 標準GUIの装飾版。
- **socket（標準ライブラリ）**: ポートスキャン・バナーグラブ・SSL確認。外部依存なしが特徴。
- **requests**: HTTPクライアント。Web探査・ファジング・認証セッション管理に使用。
- **threading / concurrent.futures**: エージェントの別スレッド実行と、スキャンの並列化（ThreadPoolExecutor）。
- **queue.Queue**: EventBus の中核（スレッドセーフなメッセージ受け渡し）。
- **openai ライブラリ**: OpenAI互換エンドポイントへの接続クライアント。
- **Pillow (PIL)**: 日本語対応PDFを画像描画で生成（[tools/pdf_writer.py](../tools/pdf_writer.py)）。フォント埋め込みの複雑さを回避する工夫。
- **NVD API v2**: CWE→CVE 照合（[tools/cve_client.py](../tools/cve_client.py)、レート制限・キャッシュ対応）。
- **LangGraph（任意）**: 監査の深層解析オーケストレーション。
- **主要ディレクトリ**: `core/`（設定・基盤）/ `agents/`（頭脳）/ `tools/`（作業）/ `gui/`（画面）/ `testlab/`（練習用脆弱サイト）/ `reports/`（成果物）。
